# eval-driven-method-fix

**Summary** — close a method gap proven by a trap scenario: minimal clause in the entry file (with compensating trims when a word budget binds), re-run of the exposing trap against the edited file as the behavioral done-signal, release with an informational migrate checklist. Fits any fix where the suite measured a hole in the shipped method rather than executor indiscipline.

## Point list

- **clause** — the minimal normative edit in the entry file; if a budget binds, compensating trims verified as relocations (greppable in their named guides), never silent deletions; rule-inventory diff pre/post. One responsibility, grep-able done-signal.
- **behavioral-validation** — re-run the exposing scenario's method arm against the edited file in a clean scratch; cap-2 against the GROUND-TRUTH anchors or the wording iterates and the point reopens. The control arm is not re-run — the fix targets the method, not the executor.
- **release** — changelog + informational migrate checklist + stamp (own edit call) + self-lint gates + run-record updated in the scenario's GROUND-TRUTH.

## Edge pattern

`clause → behavioral-validation → release` — linear; validation blocks the release whenever the wording doesn't discriminate.

## Wave shape

Single lane. Validation is the longest unit and the only one that can reopen clause. Budget the release last and alone.

## Trap warnings

- **Estimate wc, ship over budget**: measure with `wc` after the first edit pass — the clause's parenthetical cost 10 tokens the estimate missed (9 over on pass one).
- **Assumed-literal relocation checks**: greps verifying "the detail lives in X" fail on imagined literals (`L1 read-only` vs the real `L1 (report)`) — grep against the target's actual text before committing the trim.
- **Textual fixes that don't discriminate**: a clause that reads well but doesn't flip the trap is not a fix — the behavioral re-run is the done-signal, not the prose.
